# Ben Boyer — Portfolio

A fast, accessible, single-page portfolio for **Benjamin (Ben) Boyer** — an early-career
technology professional working at the intersection of IT systems, Agile delivery, cloud &
identity, and business execution.

Built from scratch with **plain HTML, CSS, and vanilla JavaScript** — no framework, no build
step, no dependencies. That keeps it fast, easy to host anywhere (GitHub Pages included), and
easy to update by hand.

---

## ✨ Features

- **Single-page layout** with smooth-scroll navigation and a scroll-spy active-section indicator
- **Dark / light theme** toggle (remembers your choice, respects system preference)
- **Animated hero** with a typewriter role rotator and a theme-aware constellation background
- **Expandable experience timeline**
- **Project case studies** in accessible modals (keyboard + screen-reader friendly)
- **Live GitHub feed** that pulls public repos from [`@bboyer23`](https://github.com/bboyer23)
- **Filterable skills** grouped by category
- **"Ben OS" layer** — a `Ctrl/Cmd + K` command palette, a **Momentum Mode** energy toggle, three
  **Recruiter / Builder / Story** viewing lenses, a **Now Building** status dashboard, a
  **Chaos → Clarity** interaction, scroll-triggered **achievement** badges, and a few easter eggs
  (type `build`, type `momentum`, or click the logo a few times) — all keyboard-accessible and
  reduced-motion aware
- **Reduced-motion aware** — all animation is disabled for users who prefer it
- **Responsive** down to small phones, with a full-screen mobile menu
- Lightweight, **no console errors**, semantic HTML, and a keyboard skip-link
- **Print-friendly** — a dedicated print stylesheet renders a clean, recruiter-ready PDF
- **Share-ready** — a custom 404 page, an XML sitemap, and rich **Open Graph / Twitter** link previews
- **Ben OS · Arcade** (`arcade.html`) — a sandbox of games written from scratch on a `<canvas>` in vanilla
  JS: **DOOMBA** (a DOOM-style raycasting FPS), Snake, and Breakout. Reachable from the footer, the ⌘K
  palette, or by typing `doom` anywhere on the site

## 🏃 Run it locally

It's static, so any local web server works. From this folder:

```bash
# Python (any version 3.x)
python -m http.server 5500
# then open http://localhost:5500
```

```bash
# or, if you prefer Node
npx serve .
```

> Opening `index.html` directly with `file://` mostly works, but a local server is recommended
> so the **live GitHub feed** and résumé/document links behave exactly like production.

There is **no build step** — what you see in the files is what ships.

## 🚀 Deploy (GitHub Pages)

This repo is already named like a Pages site (`bboyer23.github.io`). To publish:

1. Push these files to the **root** of your `bboyer23.github.io` repository (or any repo).
2. In the repo: **Settings → Pages → Build and deployment**.
3. Source: **Deploy from a branch**, Branch: **`main`** / **`/ (root)`**, then **Save**.
4. Your site goes live at `https://bboyer23.github.io/` in a minute or two.

No special configuration is needed — there's nothing to compile.

### 🔒 Keep working files & secrets out of the deploy

Everything at the repo root is served publicly. A `.gitignore` excludes the local **`Resources/`**
working folder (case-study source material, internal worklogs, an unredacted system config, and the
Worth-It app source) plus any `.env*` files. **Don't remove those ignore rules.** If the Nextcloud
admin password or any API key in that folder was ever real, rotate it — it sat in plaintext.

### ⚠️ Important: bust the cache when you change CSS/JS

Browsers (and GitHub's CDN) cache `style.css` and the `.js` files aggressively. If you edit them
and the live site looks stale or broken after deploy, it's almost always an **old cached file**.
To force every browser to fetch the new version, bump the version number in the `?v=...` query
string wherever assets are linked in `index.html` and `lab.html`:

```html
<link rel="stylesheet" href="assets/css/style.css?v=20260615b">   <!-- change to ?v=20260616, etc. -->
<script src="assets/js/main.js?v=20260615b"></script>
```

Just pick any new value (a date works well). To confirm a "broken on live" issue is caching and
not code, open the live URL in an **Incognito/Private window** or hard-refresh with **Ctrl + F5**.

> The site is also built to **never render blank if JavaScript fails to load** — an inline script
> in each page's `<head>` reveals all content as a fallback. So a cache hiccup degrades to "no
> animations," never to "blank page."

## ✏️ How to edit the content

Almost everything lives in **`index.html`**, in clearly labeled `<!-- SECTION -->` blocks.

| I want to change…            | Edit here |
|------------------------------|-----------|
| Hero headline / intro        | `index.html` → `HERO` section |

| About text & snapshot card   | `index.html` → `ABOUT` section |
| **Experience** roles/bullets | `index.html` → `EXPERIENCE (TIMELINE)` section — each role is one `<article class="tl-item">` |
| **Project cards** (the tiles)| `index.html` → `PROJECTS` section |
| **Project case studies** (modal content) | `assets/js/main.js` → the `PROJECTS` object inside `initProjectModal()` (each now uses a 5-part `spec` block: Problem / Built / Tools / Taught / Impact) |
| **Now Building** dashboard text       | `index.html` → `NOW BUILDING` section (`#now`) |
| **Chaos → Clarity** categories        | `index.html` → `APPROACH` section, `.clarity-grid` columns |
| **AI Lab preview** cards              | `index.html` → `AI LAB` section (`#ailab`) and `lab.html` |
| Command palette / view modes / Momentum copy | `assets/js/ben-os.js` (commands array) + `data-momentum-copy` attrs in `index.html` |

### Adding a new project

1. Add a card in `index.html` (copy an existing `<article class="panel project-card" ...>`),
   giving it a unique `data-project="myproject"`.
2. Add a matching entry to the `PROJECTS` object in `main.js` keyed by that same id. Each block
   supports `{ h, p }`, `{ h, ul: [...] }`, `{ h, tags: [...] }`, `{ note }`, and `{ links: [...] }`.

## 🤖 AI Lab (`lab.html`)

An optional, **access-gated** experimental demo: a small AI assistant wired to a Cloudflare
Worker that calls OpenAI. It's intentionally behind a password so the paid backend isn't abused.
It's linked from the mobile menu and the footer, not the main desktop nav.

- Backend URL is set in `assets/js/chat.js` (`WORKER_URL`).
- The access key is entered by the visitor and validated **server-side** by the Worker — it is
  not stored in this front-end code.
- If you ever retire the demo, delete `lab.html` and `assets/js/chat.js`, and remove the two
  `lab.html` links in `index.html` (mobile menu + footer).
